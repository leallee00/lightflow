# Brainstorm — 需求澄清 Skill

## 作用

在动手写代码之前，确认用户到底想要什么。
和用户聊清楚需求边界，而不是直接开始实现。

## 触发条件

- 用户发出复杂任务的指令（参考 CLAUDE.md Level 3 标准）
- 用户主动说"帮我设计一下""给个方案""你觉得怎么做"
- 需求描述模糊，关键信息缺失

## 不触发的情况

- **任务判断在阈值以下**（参考 config/threshold.md）
- 用户明确说"直接做""不用问""你自己决定"
- 只是改变量名、加注释、简单的样式修改

## 工作流程

```
1. 理解用户需求
   └─ 用自己的话复述一遍，确认理解一致
   └─ 如果用户描述清晰 → 跳转到第 3 步

2. 追问缺失的关键信息（最多 3 个问题）
   └─ 一次性问完，不要挤牙膏
   └─ 每个问题给选项（A/B/C），让用户选择
   └─ 不要问"你觉得呢"，直接给方案让用户选

3. 输出设计方案
   └─ 用 3-5 句话描述你要怎么做
   └─ 标注 1-2 个风险点
   └─ 关键：**不生成文档，不创建文件**
```

## 示例

```
用户：帮我加个用户积分系统

你（Brainstorm 模式）：
"我理解一下，你需要：
A. 简单的积分累计 + 等级展示（适合 MVP）
B. 完整的积分体系（签到、任务、兑换）
你倾向哪个？数据存在用户表里还是单独的表？"
```

## 约束

- 问题不超过 3 个
- 每个问题给选择（A/B/C），不要开放提问
- 不生成 Spec 文档
- 不创建 Git Worktree
- 聊完就停，等用户回复

## 质量检查

- [ ] 是否理解了用户的核心需求？
- [ ] 是否确认了关键边界条件？
- [ ] 是否给了用户选择而不是开放提问？
- [ ] 是否没有生成不必要的文档？
