# Review — 代码审查 Skill

## 作用

每次代码变更后，自动检查常见问题。
在用户合并代码之前，帮你发现潜在的风险。

## 触发条件

- 每次代码变更完成后自动执行（Level 1-3 都适用）
- 用户在对话中要求"帮我 review 一下"

## 不触发的情况

- 用户明确说"不用 review"
- 纯文档/注释变更
- 生成脚手架代码（初始化项目）

## 检查清单（5 个维度）

### 1. 类型安全
- [ ] 是否有 any 类型？（TypeScript 项目）
- [ ] 函数参数/返回值是否都有类型定义？
- [ ] 是否使用了 `as` 强制类型转换？（如无必要，应避免）

### 2. 错误处理
- [ ] 异步操作是否有 try/catch？
- [ ] 网络请求/数据库操作是否有超时处理？
- [ ] 错误信息是否对用户友好？（不暴露敏感信息）

### 3. 性能
- [ ] 是否有不必要的重复计算？（循环内查询、重复渲染）
- [ ] 是否有内存泄漏风险？（事件监听未清理、定时器未清除）
- [ ] 数据量大的操作是否有分页/懒加载？

### 4. 安全
- [ ] 用户输入是否经过校验/转义？
- [ ] SQL 查询是否使用参数化查询？（防注入）
- [ ] 敏感信息（密码、Token、Key）是否没有硬编码？
- [ ] 权限检查是否完备？

### 5. 测试
- [ ] 新功能是否有对应的测试？
- [ ] 边界情况是否覆盖？（空值、异常值、并发）

## 输出格式

```
## Review: [文件名]

✅ 通过 — 无问题
⚠️ 建议 — 非阻塞性建议
❌ 问题 — 必须修复

### 问题

- ❌ [文件名:行号] 问题描述
  └─ 建议修复方案

### 建议

- ⚠️ [文件名:行号] 建议描述

### 总结

发现问题 X 个，建议 Y 个。
```

## 约束

- 总输出不超过 30 行
- 没有问题时只输出 ✅ 通过，不说话
- 不修改代码，只标记问题
- 每个问题附修复建议，一句说完

## 示例

```
## Review: src/services/user.ts

### 问题
- ❌ src/services/user.ts:42 — SQL 查询使用了字符串拼接，有注入风险
  └─ 改用参数化查询: `db.query(sql, [userId])`

### 建议
- ⚠️ src/services/user.ts:88 — `userName` 变量未使用，建议移除

### 总结
发现问题 1 个，建议 1 个。
```

## 差异点

与 Superpowers 的 review 不同：
- **不读全量代码**，只检查本次变更的代码
- **不要求修改**，只标记问题
- **输出简短**，没有长分析报告
