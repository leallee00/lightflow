# Plan — 实施计划 Skill

## 作用

在动手写代码之前，列出实施步骤。
确保复杂任务被拆解为可执行的步骤序列，减少返工。

## 触发条件

- 任务涉及跨模块/跨服务改动（至少 2 个模块）
- 数据库 schema 变更（新增表、修改字段）
- API 接口重设计（新增/修改端点、变更请求/响应格式）
- 架构层面的重构（模块拆分、设计模式引入）
- 任务复杂度超过 config/threshold.md 中设定的上限

## 不触发的情况

- 单模块内的改动，不涉及外部依赖
- 纯 UI 调整（样式、布局），不涉及逻辑变更
- 简单的 CRUD 扩展（已有模式，按模板操作）

## 工作流程

```
1. 分析变更范围
   └─ 受影响文件列表
   └─ 涉及的数据流
   └─ 是否需要迁移/兼容处理

2. 输出实施步骤（3-8 步）
   └─ 每步一句话：做什么 + 在哪做
   └─ 标注步骤间的依赖关系

3. 标注风险 & 回滚方案
   └─ 如果第 X 步出问题，如何回退
   └─ 数据迁移是否可逆
```

## 输出格式

```
## 实施计划：XXX

### 步骤
1. [文件名/模块] — 做什么
2. [文件名/模块] — 做什么
   └─ 依赖：步骤 1

### 风险
- ⚠️ 风险点：XXX → 回滚方案：XXX
```

**关键约束**：输出不超过 20 行，不生成独立文件，不创建 Git Worktree。

## 示例

```
## 实施计划：添加用户积分表

### 步骤
1. database/migrations/ — 创建积分表 + 索引
2. models/UserPoints.js — 新增 model
3. controllers/userPoints.js — CRUD 接口
4. routes/ — 注册路由 + middleware
   └─ 依赖：步骤 1-3

### 风险
- ⚠️ 线上已有用户数据 → 回滚：DROP TABLE，不影响现有表
```

## 质量检查

- [ ] 步骤是否可执行？（每步都是一个明确的开发动作）
- [ ] 是否标注了依赖关系？
- [ ] 是否标注了风险点？
- [ ] 输出是否在 20 行以内？
- [ ] 没有生成独立文件？
