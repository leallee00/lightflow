# LightFlow 最佳实践

## 什么时候用完整流程

```
✅ 应该走完整流程
   ├── 新功能开发（涉及数据库/API/前端）
   ├── 架构重构
   ├── 第三方服务迁移
   ├── 安全相关改动
   └── 你不确定怎么做的时候

❌ 可以直接干
   ├── Bug fix（已知原因）
   ├── 样式/UI 调整
   ├── 重命名/重构（纯机械操作）
   ├── 添加注释/文档
   └── Code Review 意见修复
```

## 如何降低 Token 消耗

1. **保持 CLAUDE.md 简洁** — 只写项目必须的规范
2. **关闭不用的 Skill** — 在 CLAUDE.md 中注释掉
3. **设置合适的阈值** — 项目越成熟，阈值可以越高
4. **使用快速任务模板** — 参考 `templates/quick-task.md`

## 团队使用建议

- 所有成员的 CLAUDE.md 保持一致（Git 管理）
- 新成员可以先开启全部 Skill，熟悉后逐步关闭
- 代码规范写在 CLAUDE.md 中，而不是额外的 Skill 文件
- Review Skill 的检查维度可以按团队需求调整

## 常见问题

### Q: LightFlow 和 Superpowers 能一起用吗？
A: 不建议。它们的工作流可能冲突，导致 Agent 不知道该听哪个。

### Q: 支持中文吗？
A: 原生支持。所有 Skill 都会根据用户使用的语言自动切换。

### Q: 如何贡献新的 Skill？
A: 按现有 Skill 的格式写一个 `.skill.md` 文件，提 PR 即可。

### Q: 为什么每个 Skill 都强调"不生成文件"？
A: 生成 Spec/Plan 文件会占用上下文、消耗 Token、污染项目目录。LightFlow 的核心理念是「聊完就干」。
